# الجزء الأول: اللوحة المحلية

---

## ١.١ المتطلبات

- **Ruby** (تُثبَّت تلقائياً بـ `start.sh` عند الحاجة)
- **Node.js** v16 أو أحدث — https://nodejs.org
- **Git** — https://git-scm.com
- اتصال بالإنترنت (للمزامنة مع GitHub)

---

## ١.٢ التشغيل

افتح الطرفية (Terminal) وانتقل لمجلد المشروع:

```bash
cd "/مسار/GT-NEWSTECH"
bash start.sh
```

ما يحدث عند التشغيل تلقائياً:
1. **يتحقق من Ruby** — يُثبّتها إن لم تكن موجودة
2. **يُثبّت حزم Jekyll** عند أول تشغيل
3. **يُثبّت حزم Admin** عند أول تشغيل
4. **يسحب آخر التغييرات من GitHub** (مزامنة لمنع التعارض مع Decap CMS)
5. يبدأ Jekyll على `http://localhost:4000/`
6. يبدأ لوحة التحكم على `http://localhost:4001/`

افتح في المتصفح:
- الموقع: `http://localhost:4000/`
- لوحة التحكم: `http://localhost:4001/`

---

## ١.٣ مؤشر المزامنة في صفحة Git

عند الدخول لصفحة **Git / نشر** في لوحة التحكم:

**مؤشر أخضر ✓** — الجهاز محدَّث، لا توجد تغييرات جديدة من Decap CMS:
```
✓ محدّث — لا توجد تغييرات بعيدة
```

**تنبيه أصفر ⚠️** — هناك commits جديدة على GitHub لم تُسحب بعد (من Decap CMS مثلاً):
```
⚠️ المستودع البعيد متقدم بـ 2 commit — اضغط [مزامنة (Pull)] قبل التحرير
```

> **مهم:** اضغط دائماً **مزامنة (Pull)** قبل فتح أي مقال عند ظهور التنبيه الأصفر.

## ٢.٧ الخطوة السادسة — إنشاء GitHub OAuth App

1. اذهب إلى: **https://github.com/settings/developers**
   - أو من أي صفحة GitHub: انقر صورتك (أعلى اليمين) → **Settings** → (من القائمة الجانبية) **Developer settings** → **OAuth Apps**

2. انقر **New OAuth App** (أعلى اليمين)

3. أملأ النموذج بالقيم التالية بالضبط:

   | الحقل | القيمة |
   |---|---|
   | **Application name** | `GT-NEWSTECH CMS` |
   | **Homepage URL** | `https://SalehGNUTUX.github.io/GT-NEWSTECH` |
   | **Application description** | (اختياري — يمكن تركه فارغاً) |
   | **Authorization callback URL** | `https://sveltia-cms-auth.اسمك.workers.dev/callback` |

   > ⚠️ **مهم:** في **Authorization callback URL** استبدل `اسمك` برابط الـ Worker الذي حصلت عليه في الخطوة السابقة. يجب أن ينتهي الرابط بـ `/callback`

4. انقر **Register application**

5. ستظهر صفحة التطبيق. احفظ:
   - **Client ID** — ظاهر مباشرة (مثل: `Ov23xxxxxxxxxxxxx`)

6. لإنشاء الـ Client Secret:
   - انقر **Generate a new client secret**
   - أدخل كلمة مرور GitHub إذا طُلبت
   - ستظهر السلسلة مرة واحدة فقط — **انسخها فوراً** وضعها في مكان آمن

# الجزء الثالث: استخدام Decap CMS

---

## ٣.١ الدخول

1. افتح: `https://SalehGNUTUX.github.io/GT-NEWSTECH/cms/`
2. انقر **Login with GitHub**
3. إذا لم تكن مسجلاً الدخول على GitHub، أدخل بيانات حسابك
4. انقر **Authorize GT-NEWSTECH CMS** للموافقة على الصلاحيات
5. ستُعاد توجيهك تلقائياً للواجهة

---

## ٣.٢ إنشاء مقال جديد

1. من القائمة الجانبية اختر **مقالات عربية** أو **English Articles**
2. انقر **New مقال عربي** (أو **New English Article**)
3. أملأ الحقول:

   | الحقل | الوصف | مثال |
   |---|---|---|
   | **العنوان** | العنوان الكامل | `كيف تبدأ مع لينكس في 2026` |
   | **Slug** | معرّف فريد — أحرف لاتينية وشرطات فقط | `linux-getting-started-2026` |
   | **تاريخ النشر** | يأخذ اليوم تلقائياً | `2026-05-15` |
   | **القسم الرئيسي** | اختر من القائمة | `gnulinux` |
   | **أقسام إضافية** | اختياري | `foss` |
   | **الكاتب** | الاسم | `GNUTUX` |
   | **اسم ملف الصورة** | بعد رفعها (خطوة 3.3) | `linux-guide.jpg` |
   | **الملخص** | 120-160 حرف | `دليل شامل للمبتدئين...` |
   | **الوسوم** | انقر + لإضافة وسم | `لينكس`, `مبتدئين` |
   | **محتوى المقال** | الكتابة بـ Markdown | ... |

4. بعد الانتهاء انقر **Publish** (أو **Save Draft** للحفظ كمسودة)
5. يُنشئ Decap CMS commit تلقائياً على GitHub
6. GitHub Actions يبني الموقع — يظهر المقال خلال ~دقيقة

> **⚠️ قاعدة الـ Slug:** يجب أن يكون الـ slug في المقال العربي **مطابقاً تماماً** للـ slug في المقال الإنجليزي. هذا ما يربط الترجمتَين ويظهر زر "اقرأ بالعربية / Read in English".

---

## ٣.٣ رفع صورة

1. في شريط القوائم العلوي انقر **Media** (أو أيقونة الصورة)
2. انقر **Upload** واختر الصورة من جهازك
3. أو اسحب الصورة وأفلتها في المنطقة المخصصة
4. تُرفع الصورة تلقائياً للمجلد المناسب:
   - المقالات العربية → `assets/images/ar/`
   - المقالات الإنجليزية → `assets/images/en/`
5. **انسخ اسم الملف** (مثل: `linux-guide.jpg`)
6. ارجع للمقال وضع اسم الملف في حقل **اسم ملف الصورة**

> **ملاحظة:** الصيغ المرفوعة تُحفظ كما هي (لا تحويل تلقائي). استخدم اللوحة المحلية لتحويل الصور من HEIC أو BMP.

---

## ٣.٤ تعديل مقال موجود

1. من القائمة الجانبية اختر **مقالات عربية** أو **English Articles**

# الجزء الرابع: منع التعارض بين اللوحتين

---

## ٤.١ كيف يتم تجنّب التعارض تلقائياً؟

```
Decap CMS                    اللوحة المحلية
     ↓                             ↓
GitHub API                   filesystem محلي
(يقرأ من GitHub مباشرة)     (يقرأ من الجهاز)
     ↓                             ↓
          GitHub (المصدر الوحيد)
                    ↑
     مزامنة تلقائية (ff-only → rebase --autostash)
     عند تشغيل start.sh
     عند الضغط على زر "مزامنة"
     عند فشل الـ push (إعادة محاولة تلقائية)
```

- **Decap CMS** لا يملك حالة محلية — كل قراءة من GitHub مباشرة
- **اللوحة المحلية** تجري **rebase تلقائي** عند تباعد التواريخ
- التعارض الحقيقي يحدث فقط إذا عدّلت **نفس السطر في نفس الملف** من المكانين

### آلية المزامنة الذكية (محدَّثة)

عند الضغط على **مزامنة** أو عند تشغيل `start.sh`:

```
1. يجرب git pull --ff-only          (الحالة النظيفة — كل التعديلات من Decap)
   ↓ فشل؟
2. ينتقل تلقائياً لـ git pull --rebase --autostash
   ↓ يحفظ تعديلاتك غير المحفوظة (autostash)
   ↓ يعيد ترتيب commits المحلية فوق commits البعيدة
   ↓ يستعيد تعديلاتك (autostash apply)
3. إذا فشل rebase → رسالة "تعارض يحتاج حلاً يدوياً"
```

---

## ٤.٢ احتمالية تكرار المشكلة

| السيناريو | الاحتمال | الإجراء التلقائي |
|---|---|---|
| نشرت من Decap ثم فتحت اللوحة المحلية | **مرتفع** | ✅ `start.sh` يسحب تلقائياً |
| نشرت من Decap أثناء عمل اللوحة المحلية | **متوسط** | ⚠️ اضغط **مزامنة** قبل الدفع |
| نسيت المزامنة ودفعت | **منخفض** | ✅ rebase تلقائي عند الـ pull |
| نفس المقال عُدِّل في كلا اللوحتين | **نادر جداً** | ✅ Modal حل التعارض — نقرتان |
| نفس السطر في نفس الملف عُدِّل من المكانين | **استثنائي** | ✅ Modal حل التعارض |

---

## ٤.٣ قواعد العمل الآمن

| الموقف | ما تفعله |
|---|---|
| فتحت اللوحة المحلية بعد التعديل من Decap | ✅ `start.sh` سيسحب تلقائياً |
| اللوحة المحلية تعمل وتريد التبديل لـ Decap | أنهِ وادفع (`Commit & Push`) أولاً |
| رأيت التنبيه الأصفر في صفحة Git | اضغط **مزامنة (Pull)** قبل فتح أي مقال |
| تعديل نفس المقال من كلا اللوحتين | ❌ تجنب — اختر واحدة فقط |
| عدة أشخاص يعملون في نفس الوقت | ✅ مقبول بشرط عدم تعديل نفس الملف |

### نصائح لتقليل الحاجة للـ rebase

- **اضغط "مزامنة"** قبل بدء أي جلسة تحرير في اللوحة المحلية
- **اضغط "Commit & Push"** فور انتهائك من تعديل — لا تترك commits معلَّقة طويلاً
- **في Decap CMS**: انشر (Publish) فوراً عند الانتهاء — لا تترك Drafts معلَّقة

---

## ٤.٤ ماذا لو حدث تعارض حقيقي؟

النظام يحاول الحل التلقائي بـ rebase. إذا فشل (نفس السطر عُدِّل من المكانين)، **تظهر نافذة حل التعارض تلقائياً** — لا حاجة لفتح طرفية.

### الواجهة شبه الآلية (الموصى بها)

عند فشل المزامنة بسبب تعارض، يظهر Modal مباشرة:

```
⚠️  حل تعارض في 2 ملف
─────────────────────────────────
📄 _ar/foss/article.md         ⏳ غير محلول
   ┌─ 💻 نسختي المحلية ──┐  ┌─ ☁️ نسخة Decap CMS ─┐
   │ (معاينة 600 حرف)    │  │ (معاينة 600 حرف)    │
   └─────────────────────┘  └─────────────────────┘
   [احتفظ بنسختي]  [احتفظ بنسخة GitHub]

📄 _en/foss/article.md         ⏳ غير محلول
   [...]

تم حل 0 من 2
─────────────────────────────────
[إكمال الدمج والدفع]    [إلغاء المزامنة]
```

**كيف يعمل:**
1. تنقر **"احتفظ بنسختي"** أو **"احتفظ بنسخة GitHub"** لكل ملف
2. الحالة تتحول إلى ✓ والنسخة المختارة تُبرز بإطار ذهبي
3. عند حل كل الملفات → زر **"إكمال الدمج والدفع"** يُفعَّل
4. نقرة واحدة → `rebase --continue` + `push` تلقائياً

**ميزات إضافية:**
- إذا ظهرت تعارضات إضافية من commit آخر (في rebase متعدد) → Modal جديد يفتح تلقائياً
- **زر "إلغاء المزامنة"** يستدعي `rebase --abort` ويعيدك لحالة ما قبل المحاولة بأمان

### الحل اليدوي (احتياطي فقط)

إذا أردت دمج النسختين معاً يدوياً (لا اختيار واحدة كاملة):

# افتح الملف المتعارض وابحث عن:
#    <<<<<<< HEAD

#    =======
#    (النسخة البعيدة من Decap)
#    >>>>>>> origin/main
# احذف العلامات واترك ما تريد (يمكنك دمج النصين)

git add .
git rebase --continue
git push origin main

# إلغاء كلياً:
git rebase --abort
```

> للمزيد من التفاصيل راجع: `دليل-الربط-بـGitHub.md`

### كيف يعمل في الواجهة

بعد تفعيله، سيظهر شريط جديد في CMS بثلاثة أعمدة:

┌──────────────┬──────────────┬──────────────┐
│ ✏️  مسودة     │ 👀 مراجعة    │ ✅ جاهز      │

```

- In Review → Pull Request مفتوح
- Ready → جاهز للدمج في main

## ٧.٧ المقال يظهر في القائمة لكن يعطي 404 عند الدخول

### السبب الأكثر شيوعاً تاريخياً (محلول جذرياً)

**التفسير:** خادم اللوحة المحلية كان يستخدم `Date.UTC(...)` لتحويل التاريخ، وهو يعامل إدخال المستخدم كأنه **UTC أصلاً**. لكن المستخدم يُدخل بتوقيته المحلي.

**النتيجة الكارثية:**

| موقع المستخدم | يُدخل محلياً | كان يُحفظ خطأً | الفعلي UTC الآن | الفرق |
|---|---|---|---|---|
| الجزائر (UTC+1) | 14:00 | 14:00 UTC | 13:00 UTC | +1 ساعة في المستقبل |
| الصين (UTC+8) | 14:00 | 14:00 UTC | 06:00 UTC | +8 ساعات في المستقبل |
| كاليفورنيا (UTC-8) | 14:00 | 14:00 UTC | 22:00 UTC | -8 ساعات (لا مشكلة) |

Jekyll يتخطّى المقالات المستقبلية افتراضياً → **بطاقة في القائمة + لا صفحة = 404**.

### الإصلاح الذي تم تطبيقه

في `admin/server.js` (`saveArticle`):
```javascript
// قبل (خطأ):
fm.date = new Date(Date.UTC(y, m-1, d, h, min, s));

// بعد (صحيح):
fm.date = new Date(y, m-1, d, h, min, s);
```

**الفرق:** `new Date(...)` بدون `UTC` يفسّر الأرقام بتوقيت النظام، ثم Date يُحوّلها داخلياً لـ UTC. النتيجة = اللحظة الحقيقية المطلقة، يعمل من أي بلد.

**شبكة أمان إضافية:** `_config.yml` يحتوي على `future: true` لتجاوز أي تأخير زمني نادر.

### إذا واجهت 404 رغم ذلك

**أ) GitHub Actions لم يكتمل بعد:**
1. انتظر ~دقيقة كاملة
2. تحقق من: https://github.com/SalehGNUTUX/GT-NEWSTECH/actions
3. أعد فتح الرابط بـ **Ctrl+Shift+R**

**ب) خطأ في front matter أو Liquid syntax:**
```bash
cd /مسار/GT-NEWSTECH
bundle exec jekyll build --config _config.yml,_config.local.yml
find _site -name "*المقال*"
```
إذا لم تُولَّد الصفحة محلياً → خطأ في المقال نفسه (راجع رسائل البناء).

**ج) مقال قديم بتاريخ محفوظ خطأً قبل الإصلاح:**
افتح الملف وأصلح حقل `date` يدوياً ليكون قبل الوقت الحالي UTC:
```yaml
# مثال: لو الآن 14:00 UTC، اجعل التاريخ:
date: 2026-05-12T13:00:00.000Z
```

## ٧.٨ خطأ "Updates were rejected" أو "non-fast-forward" عند الدفع

**السبب:** Decap CMS نشر تعديلات لم تُسحب بعد للجهاز المحلي.

**الحل التلقائي (الموصى به):**
- في لوحة التحكم → **Git** → اضغط **مزامنة (Pull)**
- النظام يجرّب `ff-only` ثم ينتقل تلقائياً لـ `rebase --autostash`
- بعد المزامنة → اضغط **Commit & Push**

**إذا ظهر تعارض حقيقي:** Modal "حل التعارض" يفتح تلقائياً — راجع §٤.٤ لتفاصيل الواجهة.

## ٨.١ تحديث قائمة الأقسام في Decap CMS

عند إضافة قسم جديد من اللوحة المحلية، يجب تحديث `cms/config.yml` يدوياً.

ابحث عن قسمَي `options:` (في كل من AR و EN collections) وأضف:

```yaml
- { label: "اسم القسم الجديد", value: new-category-id }
```

مثال لإضافة قسم **الأجهزة**:

```yaml
options:
  - { label: "مشاريع GNUTUX",    value: gnutux-projects }
  - { label: "البرمجيات الحرة",  value: foss }
  - { label: "غنو/لينكس",       value: gnulinux }
  - { label: "أخبار التقنية",    value: tech-news }
  - { label: "الذكاء الاصطناعي",value: ai }
  - { label: "ألعاب لينكس",      value: gaming }
  - { label: "الأجهزة",          value: hardware }    # ← أضف هنا
```

ثم ارفع:
```bash
git add cms/config.yml
git commit -m "config: إضافة قسم الأجهزة لـ Decap CMS"
git push origin main
```

## ٨.٢ تحديث Decap CMS لأحدث إصدار

في `cms/index.html` غيّر:
```html
<!-- من: -->
<script src="https://unpkg.com/decap-cms@^3.0.0/dist/decap-cms.js"></script>

<!-- إلى (مثال للإصدار الجديد): -->
<script src="https://unpkg.com/decap-cms@^3.1.0/dist/decap-cms.js"></script>
```

---

## ٨.٣ تجديد Cloudflare Worker (إذا تغيّر الرابط)

إذا احتجت إعادة نشر Worker بعنوان مختلف:
1. عدّل `cms/config.yml` بالرابط الجديد
2. عدّل **Authorization callback URL** في GitHub OAuth App
3. ارفع `cms/config.yml`
