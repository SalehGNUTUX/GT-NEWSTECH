## الجزء الأول: اللوحة المحلية

### التشغيل

```bash
bash start.sh
```

عند التشغيل:
1. يسحب تلقائياً أحدث التغييرات من GitHub (مزامنة)
2. يفتح Jekyll على `localhost:4000`
3. يفتح لوحة التحكم على `localhost:4001`

### مؤشر المزامنة في صفحة Git

**تنبيه أصفر (مزامنة مطلوبة):** يظهر عندما يكون المستودع البعيد متقدماً بـ commits — اضغط **مزامنة (Pull)** قبل التحرير.

**مؤشر أخضر:** الجهاز محدَّث، لا توجد تغييرات من Decap CMS.

## الجزء الثاني: إعداد Decap CMS

### الخطوة 1 — إنشاء GitHub OAuth App

1. اذهب إلى: **github.com → Settings → Developer settings → OAuth Apps → New OAuth App**
2. أدخل المعلومات:
   ```
   Application name: GT-NEWSTECH CMS
   Homepage URL: https://SalehGNUTUX.github.io/GT-NEWSTECH
   Authorization callback URL: https://YOUR-WORKER.workers.dev/callback
   ```
3. اضغط **Register application**
4. احفظ: **Client ID** و **Client Secret**

> ⚠️ Client Secret يظهر مرة واحدة فقط — احفظه فوراً

---

### الخطوة 2 — نشر OAuth Proxy على Cloudflare Workers

> هذا الـ proxy يتوسط بين Decap CMS وGitHub OAuth بشكل آمن

**أ) أنشئ حساباً على Cloudflare** (مجاني): cloudflare.com

**ب) ثبّت Wrangler**:
```bash
npm install -g wrangler
wrangler login
```

**ج) استنسخ Sveltia CMS Auth**:
```bash
git clone https://github.com/sveltia/sveltia-cms-auth.git
cd sveltia-cms-auth
npm install
```

**د) أضف بيانات GitHub OAuth**:
```bash
wrangler secret put GITHUB_CLIENT_ID
# أدخل Client ID عند الطلب

wrangler secret put GITHUB_CLIENT_SECRET
# أدخل Client Secret عند الطلب
```

**ه) انشر الـ Worker**:
```bash
wrangler deploy
```

ستحصل على رابط مثل:
```
https://sveltia-cms-auth.YOUR-NAME.workers.dev
```

## الجزء الثالث: استخدام Decap CMS

### الدخول

1. افتح `https://SalehGNUTUX.github.io/GT-NEWSTECH/cms/`
2. اضغط **Login with GitHub**
3. أقبل الصلاحيات (قراءة/كتابة المستودع)

### إنشاء مقال جديد

1. اختر **مقالات عربية** أو **English Articles**
2. اضغط **New مقال عربي**
3. أملأ الحقول:
   - **Slug**: أحرف لاتينية وشرطات — **نفسه بالضبط** في نظيره الآخر
   - **القسم**: اختر من القائمة
   - **التاريخ**: يأخذ اليوم الحالي تلقائياً
4. اضغط **Publish** — يُنشئ commit تلقائياً

### رفع صورة

1. انقر **Media** في القائمة العلوية
2. اسحب الصورة أو انقر للاختيار
3. الصورة تُرفع لـ `assets/images/ar/` (أو `en/`)
4. في حقل **اسم ملف الصورة**: اكتب اسم الملف يدوياً (مثال: `my-image.jpg`)

> ملاحظة: الصور تُرفع دائماً للمجلد المحدد في الإعدادات. غيّر الـ collection المفتوح حسب اللغة قبل الرفع.

## الجزء الرابع: منع التعارض بين اللوحتين

### لماذا لا يحدث تعارض؟

```
Decap CMS                    اللوحة المحلية
     ↓                             ↓
GitHub API                   filesystem
(دائماً آخر commit)          (ملفات محلية)
     ↓                             ↓
          GitHub (المصدر الوحيد)
                    ↑
              git pull تلقائي
              عند تشغيل start.sh
```

- **Decap CMS** لا يحتفظ بحالة محلية — يقرأ دائماً من GitHub
- **اللوحة المحلية** تسحب تلقائياً عند التشغيل

### قواعد العمل الآمن

| الموقف | ماذا تفعل |
|---|---|
| بدأت بالتحرير في Decap من الهاتف | أكمل وانشر قبل التبديل للمحلي |
| ستفتح اللوحة المحلية بعد Decap | تأكد أن `start.sh` سحب التغييرات (الرسالة الخضراء) |
| اللوحة المحلية تُظهر تنبيه المزامنة | اضغط **مزامنة (Pull)** قبل فتح أي مقال |
| تحرير نفس المقال من كلا اللوحتين | ❌ تجنب ذلك دائماً |

### ماذا لو حدث تعارض؟

```bash
cd /مسار/GT-NEWSTECH

# عرض حالة التعارض
git status

# اختيار 1: احتفظ بنسخة GitHub (تتجاهل التغييرات المحلية)
git checkout -- FILENAME

# اختيار 2: احتفظ بالتغييرات المحلية (تتجاهل GitHub)
git checkout FILENAME
git add . && git commit -m "fix: حل تعارض" && git push
```

## الجزء السادس: صيانة دورية

### تحديث قائمة الأقسام في Decap CMS

عند إضافة قسم جديد من اللوحة المحلية، يجب تحديث `cms/config.yml` يدوياً:

```yaml
options:
  - { label: "القسم الجديد", value: new-category }
```

أضفه في كلتا مجموعتَي AR و EN.

### مشكلة التواريخ

مقالات تم إنشاؤها عبر Decap CMS تُخزَّن بتاريخ بدون وقت (`2026-05-15`). هذا مقبول لكن يعني أن الترتيب داخل نفس اليوم غير محدد. لتحديد الترتيب بدقة، افتح المقال من اللوحة المحلية وحدد الوقت.
